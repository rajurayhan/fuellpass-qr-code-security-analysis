Security Analysis / Fuellpass QR
April 2026 — Bangladesh
Critical infrastructure security

QR code security in
Fuellpass

A deep dive into how Fuellpass encodes user identity in QR tokens, where the current design falls short, and how a layered online/offline architecture can close the real-world attack surface.

What the QR actually contains

Scanning a Fuellpass QR yields a two-part dot-separated string. The first part is a Base64-encoded JSON payload. The second is a cryptographic signature.

{
  "uid": 37626,
  "phone": "01849699001",
  "iat":   1775802947,
  "created_at": 1775802947
}

The signature — a 64-character hex string — is computed with HMAC-SHA256. The payload is readable by anyone but supposedly tamper-proof without the server's secret key.

How HMAC-SHA256 signing works

The server computes HMAC_SHA256(secret_key, payload) and appends it. A verifier recomputes the same hash — if it matches, the token is genuine. Any change to the payload breaks the hash.

The obvious attack: payload spoofing

A naive attacker might modify the payload, re-encode it, and reuse the original signature. Here's what that looks like.

decode payload
→
modify uid / phone
→
re-encode base64
→
keep old signature
→
new QR
This attack fails — if implemented correctly

HMAC-SHA256 is computed against the original payload. A tampered payload produces a different hash. The scanner rejects it. Cryptographically sound.

But that "if" matters. Common implementation mistakes can render the signing useless.

Critical
Signature never verified

Token is generated with a signature but the scanner reads the payload blindly without checking it.

Critical
Weak or hardcoded secret

A guessable key like secret123 lets an attacker brute-force and re-sign any payload they want.

High
Algorithm confusion

If the system accepts "alg": "none", the signature can be stripped entirely — a classic JWT flaw.

High
No expiry enforced

iat is logged but never checked against an expiry window. A token remains valid forever.

The real threat: replay attacks

Payload spoofing requires breaking cryptography. A replay attack requires only a camera. The attacker doesn't forge anything — they simply reuse a valid QR they photographed.

Scenario — Dhaka CNG station, peak hour You open Fuellpass and show your QR to the attendant. Someone behind you in the queue casually films your phone screen. Ten minutes later, at a different pump, they scan your QR. Fuel is dispensed from your quota. You find out only when your monthly allocation runs out — with no explanation, no alert, no recourse.

This is possible because the current token design has structural weaknesses that make replaying a captured QR trivially easy.

Bottom line

The HMAC signature is cryptographically sound. But a valid signature on a static, long-lived token is not meaningful security. A photograph of your QR is as good as your physical presence at any pump — potentially forever.

A better design: layered online/offline tokens

The natural fix is a per-transaction QR — generated fresh on the server for every fuel request, single-use, expiring in 60 seconds. This is how bKash, Nagad, and Google Pay handle it. It closes the replay attack completely.

But Bangladesh has a real connectivity problem. Rural areas, congested networks, and server downtime mean you cannot require internet at the moment of dispensing. A purely online system fails users when they need it most.

The solution is a two-mode architecture.

Online mode — per-transaction QR

User requests fuel in app

Server generates single-use QR

QR expires in 60 seconds

Server marks nonce as used on scan

Duplicate scan returns error instantly

Offline mode — backup token

Pre-generated when user last had internet

Two time-sliced single-use tokens

Station verifies signature locally

Usage synced to server on reconnect

Used nonces invalidated on sync

Offline backup token structure

Instead of a use-counter (which can be manipulated on a rooted device), the server pre-generates two tokens covering different time windows.

{
  "uid": 37626,
  "phone": "01849699001",
  "type": "backup",
  "issued_at": 1775802947,
  "exp":  1775889347,       // valid 24 hours total
  "window": "0-12h",        // only active in first half
  "nonce": "x9f2a1c8e3b74d05"
}

Only one token is mathematically active at any moment. Even if a hacker photographs both simultaneously, they can only exploit one 12-hour window — and your server can flag duplicate nonce attempts when the device reconnects.

Risky
Device-side counter

App tracks uses_remaining = 2 locally. Fails if the app is tampered with or device is rooted. Counter can simply be reset.

Better
Time-sliced tokens

No counter to manipulate. Only one token is valid at any given time. Damage window is bounded to 12 hours maximum.

Recommended safeguards

1
Dynamic QR with short expiry
Online QR should expire in 30–60 seconds. Add an exp field and enforce it strictly at the scanner. If the token is older than the window, reject unconditionally.
2
One-time nonce
Every token carries a cryptographically random nonce. The server marks it used on first scan. A second scan of the same nonce is rejected with an error — no matter how valid the signature is.
3
Biometric gate on backup QR
The offline backup QR should sit behind fingerprint or PIN in the app. Prevents casual misuse if the phone is stolen or borrowed.
4
Instant SMS notification on every scan
Every QR scan — online or offline — triggers an SMS to the registered mobile number. Users catch unauthorized use in real time, even without a data connection.
5
Server reconciliation on reconnect
When the user's device reconnects, the app syncs offline scan events. The server checks for duplicate nonces across all stations and flags anomalies for investigation.
6
Station geo-log
Offline scans should record station GPS coordinates locally and sync them on reconnect. If the same token appears at two geographically distant stations, it is flagged automatically.

Key takeaway

The HMAC-SHA256 signing on Fuellpass tokens is a solid foundation. But a valid signature on a static, long-lived token is not enough. Short expiry, one-time nonces, time-sliced offline tokens, and real-time SMS alerts together close the replay attack surface — without breaking the offline usability Bangladeshi users genuinely need.

This analysis is based on a QR token captured in the field. Internal implementation details of Fuellpass's backend are not known to the author. Any identified vulnerabilities should be reported responsibly to the Fuellpass security team before public disclosure.
